# WA-018: ES 7.x Mapping Type Removal (Storefront & Admin)

**Date:** 2026-02-12  
**Status:** Complete  
**Branch:** modernization/upgrade-platform-2026-02  
**Related:** WA-017 (ES 7.x Mapping Type Removal - Core)

## Summary

Verified that storefront/ and admin/ directories require **no ES 7.x mapping type changes**. All Elasticsearch mapping definitions, document operations, and query building are handled in the `core/` directory, which was already fixed in WA-017.

## Investigation

Searched storefront/ and admin/ for the same ES 7.x incompatible patterns that were fixed in WA-017:

### Patterns Searched
1. **Mapping type wrappers** - nested under type name like `{ storefront: { properties: ... } }`
2. **`type: 'string'` fields** - should be `text` or `keyword`
3. **`not_analyzed`** - on string fields
4. **Dynamic type parameters** - in document operations (should be `type: '_doc'`)
5. **MLT query syntax** - `ids` parameter (should be `like` with document refs)
6. **Total count format** - integer vs `{ value: N }` handling

### Findings

**No ES-specific code found in storefront/ or admin/ that requires changes:**

- ✅ No mapping definitions (all in `core/lib/workarea/configuration.rb`)
- ✅ No direct ES index operations (all through `core/lib/workarea/elasticsearch/document.rb`)
- ✅ No custom query building (all in `core/app/queries/workarea/search/`)
- ✅ No direct response parsing that needs total count format handling

**What exists in storefront/ and admin/:**
- View models that wrap `Search::StorefrontSearch` and `Search::AdminSearch` (from core)
- Controllers that use search classes (from core)
- Tests that use search functionality (but don't define mappings or do direct ES ops)

All Elasticsearch interaction is abstracted through core classes, which were already fixed in WA-017.

## Incidental Fix

Found and fixed a **Logger boot order issue** in admin test environment (not ES-related):

**File:** `admin/test/dummy/config/boot.rb`

**Problem:** Ruby 2.6+ with Rails 6.1 can hit `uninitialized constant ...::Logger` if stdlib Logger isn't loaded before ActiveSupport initializes.

**Fix Applied:** Added `require 'logger'` before ActiveSupport loads, matching the fix previously applied to storefront in commit `117cff49`.

```ruby
# Set up gems listed in the Gemfile.
ENV['BUNDLE_GEMFILE'] ||= File.expand_path('../../../../Gemfile', __dir__)

require 'bundler/setup' if File.exist?(ENV['BUNDLE_GEMFILE'])

# Ruby 2.6+ + Rails 6.1 can hit `uninitialized constant ...::Logger` if stdlib
# Logger isn't loaded before ActiveSupport initializes.
require 'logger'

$:.unshift File.expand_path('../../../../lib', __dir__)
```

## Architecture Notes

Workarea's Elasticsearch integration follows a clean separation of concerns:

- **Core (`core/`)**: All ES mappings, indexing, and query logic
  - `lib/workarea/configuration.rb` - Mapping definitions
  - `lib/workarea/elasticsearch/document.rb` - Index operations
  - `lib/workarea/elasticsearch/index.rb` - Low-level index management
  - `app/queries/workarea/search/` - Query builders
  - `app/models/workarea/search/` - Search models

- **Storefront (`storefront/`)**: UI and view models only
  - No direct ES operations
  - Uses `Search::StorefrontSearch` from core

- **Admin (`admin/`)**: UI and view models only
  - No direct ES operations
  - Uses `Search::AdminSearch` from core

This architecture means ES 7.x compatibility changes only need to be made in core, not in storefront or admin.

## Conclusion

**No ES 7.x mapping type changes required for storefront/ or admin/.**

All Elasticsearch code is centralized in core, which was already fixed in WA-017. The storefront and admin directories only contain UI layer code that consumes search functionality through core's APIs.

The only change made was fixing the Logger boot order issue in admin tests, which is unrelated to ES but was blocking test execution.
