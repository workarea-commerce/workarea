# WA-011: Performance Test Suite - Elasticsearch 7.x Compatibility

**Date:** 2026-02-12  
**Status:** In Progress - Multiple ES 7.x compatibility issues identified and partially fixed  
**Branch:** modernization/upgrade-platform-2026-02  
**Environment:** Elasticsearch 7.17.0, Ruby 2.7.8

## Summary

Attempted to run the performance test suite (`bundle exec rake performance_test`) on the platform upgrade branch. The suite revealed significant Elasticsearch 7.x compatibility issues that need to be addressed. Several fixes have been implemented, but additional work is required.

## Elasticsearch Version Mismatch

The codebase uses:
- **Server:** Elasticsearch 7.17.0
- **Client:** elasticsearch-api 5.0.5, elasticsearch-transport 5.0.5

This creates compatibility issues because:
1. The client library was designed for ES 5.x/6.x
2. ES 7.x removed mapping types and changed several API formats
3. ES 7.x changed response structures (e.g., `hits.total` format)

## Issues Identified & Fixed

### 1. ✅ Elasticsearch Mapping Type Wrappers
**Problem:** Mappings used old nested type structure incompatible with ES 7.x  
**File:** `core/lib/workarea/configuration.rb`  
**Fix:** Removed type wrappers (`admin`, `storefront`, `help`, `order`) from mapping definitions

**Before:**
```ruby
config.elasticsearch_mappings.admin = {
  admin: {
    dynamic_templates: [...],
    properties: {...}
  }
}
```

**After:**
```ruby
config.elasticsearch_mappings.admin = {
  dynamic_templates: [...],
  properties: {...}
}
```

###2. ✅ Deprecated Field Types
**Problem:** `type: 'string'` and `index: 'not_analyzed'` deprecated in ES 7.x  
**Files:** `core/lib/workarea/configuration.rb`  
**Fix:** Updated to use `type: 'text'` or `type: 'keyword'`

### 3. ✅ Document Type Parameters
**Problem:** Code passing `type: type` to ES operations, but ES 7.x only uses `_doc`  
**File:** `core/lib/workarea/elasticsearch/document.rb`  
**Fix:** Changed all ES operations to use `type: '_doc'` instead of dynamic types

### 4. ✅ Total Count Format Change
**Problem:** ES 7.x changed `hits.total` from integer to `{value: N, relation: "eq"}`  
**Files Fixed:**
- `core/lib/workarea/paged_array.rb`
- `core/lib/workarea/elasticsearch/index.rb`
- `core/app/queries/workarea/search/query.rb`
- `core/app/queries/workarea/search/storefront_search/product_multipass.rb`

**Fix:** Added logic to handle both formats:
```ruby
total = total.is_a?(Hash) ? total['value'] : total
```

### 5. ✅ Deprecated Query Parameter
**Problem:** `minimum_number_should_match` renamed to `minimum_should_match` in ES 7.x  
**File:** `core/app/queries/workarea/search/product_search.rb`  
**Fix:** Updated field name

### 6. ✅ Category Query Type Parameters
**Problem:** Explicit `type: 'category'` being passed in save/delete operations  
**File:** `core/app/models/workarea/search/storefront/category_query.rb`  
**Fix:** Removed explicit type parameters (relies on `_doc` from document.rb)

## Issues Still Requiring Fixes

### 7. ⚠️ Direct Index Operations Require Type Parameter
**Error:** `Required argument 'type' missing` in save/delete/update/search/count operations  
**Root Cause:** The `Workarea::Elasticsearch::Index` class does not add a default `type` parameter, but the 5.x elasticsearch-ruby client requires it for these operations.  
**Impact:** ANY direct call to `Index#save`, `Index#delete`, `Index#update`, `Index#search`, or `Index#count` will fail unless `type: '_doc'` is explicitly passed.

**Example Files Affected:**
- `core/app/models/workarea/search/storefront/category_query.rb` - both `save` and `delete` operations

**Required Fix:** Add `type: '_doc'` parameter to all direct Index operations
```ruby
# Both save and delete need the type parameter
Storefront.current_index.save(document, type: '_doc')
Storefront.current_index.delete(category.id, type: '_doc')
```

**Note:** This is a temporary workaround for the 5.x client. When upgrading to elasticsearch-ruby 7.x, the client will handle `_doc` as the default type and these explicit parameters can be removed.

### 8. ⚠️ More Like This (MLT) Query Syntax
**Error:** `[mlt] query does not support [ids]`  
**Impact:** Recommendations feature failing  
**Files Affected:**
- `core/app/queries/workarea/search/related_products.rb`
- `core/app/queries/workarea/search/related_help.rb`

**Root Cause:** ES 7.x removed the `ids` parameter from MLT queries

**Required Fix:** Update MLT query syntax to use `like` parameter with document structure instead of `ids`

**ES 5.x/6.x syntax (current, broken):**
```ruby
more_like_this: {
  ids: ['product_id_1', 'product_id_2'],
  like_text: 'some text',
  # ... other params
}
```

**ES 7.x compatible syntax (target):**
```ruby
more_like_this: {
  like: [
    { _id: 'product_id_1' },
    { _id: 'product_id_2' }
  ],
  # ... other params (like_text should be migrated to 'like' array as well)
}
```

**Note:** The `like` parameter accepts an array of document references. Each reference can be either a document structure (`{_id: ...}`) or a text string.

### 9. ⚠️ Keyword Field Analyzer Incompatibility Risk
**Location:** `core/lib/workarea/configuration.rb` - admin mapping dynamic template  
**Issue:** The admin mapping includes `type: 'keyword'` with `analyzer: 'keyword'`
```ruby
mapping: { type: 'keyword', analyzer: 'keyword' }
```

**Risk:** In ES 7.x, `keyword` fields generally don't accept the `analyzer` parameter (they use `normalizer` instead). Depending on ES strictness and client behavior, this may produce a mapping error during index creation.

**Required Action:** Either:
- Verify this mapping successfully creates in ES 7.17 (test and document outcome), **or**
- Remove the `analyzer` parameter from keyword fields, **or**
- Replace `analyzer: 'keyword'` with `normalizer: 'keyword_normalizer'` (if a normalizer is defined)

**Status:** Unverified - needs testing during next index creation/refresh
