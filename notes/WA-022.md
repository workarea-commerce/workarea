# WA-022: Fix sysctl PATH issue in test environment

## Problem
Some system tests that process images (e.g. storefront tests that create assets) can fail with:

```
Errno::ENOENT: No such file or directory - sysctl
```

`image_optim` shells out to `sysctl` to detect processor count. In certain CI / sandboxed environments, the default `PATH` omits `/usr/sbin`, so `sysctl` can’t be found even though it exists at `/usr/sbin/sysctl`.

## Fix
In the Workarea core engine, when `Rails.env.test?`, ensure `/usr/sbin` is present in `ENV['PATH']` (if `/usr/sbin/sysctl` exists). This keeps the change scoped to the test environment and avoids modifying application behavior in development/production.

Implementation:
- `core/lib/workarea/core/engine.rb`
  - In the existing `Rails.env.test?` block, prepend `/usr/sbin` to `ENV['PATH']`.

## How to verify
With a restricted PATH that omits `/usr/sbin`, boot the dummy app in test env and instantiate ImageOptim:

```bash
cd storefront/test/dummy
RAILS_ENV=test PATH="$HOME/.rbenv/shims:$HOME/.rbenv/bin:/usr/bin:/bin" bin/rails runner "require 'image_optim'; ImageOptim.new; puts 'ok'"
```

Expected:
- Prints `ok`
- No `sysctl` ENOENT raised

## Notes
You may still see warnings about missing optional ImageOptim workers (e.g. `pngout`, `svgo`) depending on local binaries; these are unrelated to the `sysctl` PATH failure.
